# Checklist ก่อน deploy

บทสุดท้ายใช้ตรวจว่า API พร้อมส่งงานหรือ deploy แล้วหรือยัง

ให้ใช้ checklist นี้ก่อน push code, เปิด pull request หรือส่งโปรเจกต์เป็น portfolio

## วิธีเรียนบทนี้

ให้ตรวจทีละหมวด:

1. ถ้าข้อไหนยังไม่ผ่าน ให้แก้ก่อน
2. ถ้าข้อไหนยังไม่เกี่ยวกับโปรเจกต์ ให้บันทึกว่า "not applicable"
3. ถ้าข้อไหนเป็นความเสี่ยงที่ยอมรับได้ ให้เขียนเหตุผลไว้ใน README หรือ release note
4. อย่าใช้คำว่า production ready ถ้ายังมี secret ใน Git หรือ test/build ไม่ผ่าน

Checklist ไม่ได้ทำให้ระบบปลอดภัยอัตโนมัติ แต่ช่วยลดการลืมเรื่องพื้นฐานก่อน deploy

## สิ่งที่จะใช้ในบทนี้

| สิ่งที่จะใช้ | ความหมาย |
| --- | --- |
| release checklist | รายการตรวจซ้ำก่อนส่งงานหรือ deploy |
| smoke test | test สั้น ๆ หลังรัน app จริง |
| secret review | ตรวจว่าไม่มี secret หลุดเข้า Git |
| migration review | ตรวจว่า database schema พร้อมสำหรับ environment ปลายทาง |
| deployment note | บันทึกข้อจำกัดหรือความเสี่ยงที่ยังเหลือ |

## Code และ Build

- `dotnet build -c Release` ผ่าน
- `dotnet test -c Release` ผ่าน
- ไม่มี warning สำคัญที่ถูก ignore โดยไม่มีเหตุผล
- ไม่มี code debug ชั่วคราว เช่น temporary token หรือ `Console.WriteLine` ที่ไม่จำเป็น
- ไม่มี endpoint ทดสอบที่เปิด public โดยไม่ตั้งใจ
- code ที่เกี่ยวกับ auth/admin มี test ครอบคลุม case สำคัญ

## Configuration

- ไม่มี production secret ใน Git
- `Jwt:SigningKey` มาจาก environment variable หรือ secret store
- connection string production ไม่อยู่ใน `appsettings.json`
- `ASPNETCORE_ENVIRONMENT` ถูกตั้งถูก environment
- app fail ตอน start ถ้า config สำคัญหาย
- `.env` อยู่ใน `.gitignore`
- `.env.example` มีเฉพาะ key ตัวอย่างหรือ placeholder

## Security

- password ถูก hash ก่อนบันทึก database
- response ไม่ส่ง `PasswordHash`
- JWT signing key ยาวและสุ่มพอ
- endpoint ที่ต้อง login ใช้ `[Authorize]`
- admin endpoint ใช้ `[Authorize(Roles = Roles.Admin)]`
- admin ทำ self-deactivate หรือ self-demote ไม่ได้
- error response ไม่แสดง stack trace
- log ไม่เก็บ password, token, signing key หรือ secret

## Database

- migration ล่าสุดถูกสร้างแล้ว
- migration ถูกทดสอบกับ database local หรือ test database
- index สำคัญรองรับ query ที่ใช้จริง เช่น email, token hash, refresh token family, audit log และ admin user list
- มีแผนรัน migration ใน environment ปลายทาง
- seed data production ไม่ใช้รหัสผ่านตัวอย่าง
- audit log ไม่เก็บ secret
- backup/restore database ถูกวางแผนสำหรับระบบจริง

## API Contract

- OpenAPI document เปิดดูได้ใน environment ที่ต้องการ
- response status code สอดคล้องกับบทที่ 27
- validation error มี `code = VALIDATION_FAILED`
- business error ใช้ `ProblemDetails`
- endpoint สำคัญมี `ProducesResponseType`
- route ที่เปลี่ยนแล้วถูกอัปเดตใน `.http`, README และเอกสาร

## Observability

- มี structured logging ใน auth และ admin action
- log ไม่เก็บ password, token หรือ secret
- error log มี trace id หรือข้อมูลพอใช้ค้นต่อ
- log level ใน production ไม่ละเอียดเกินไป
- audit log มี event สำคัญสำหรับ admin action

## Docker และ Deploy

- `Dockerfile` build ผ่าน
- `docker compose config` ผ่าน
- `docker compose up --build` ผ่าน
- image runtime ใช้ `aspnet` ไม่ใช่ `sdk`
- container runtime ไม่รันด้วย root user
- `.dockerignore` กัน `.env`, build output และไฟล์ไม่จำเป็นออกจาก build context
- container รับ config ผ่าน environment variables
- port mapping ถูกต้อง
- smoke test หลังรัน container ผ่าน

## สิ่งที่ควรต่อยอดหลังจบหนังสือ

- email provider จริงสำหรับ forgot password และ email confirmation
- MFA สำหรับ admin
- API versioning
- centralized logging backend
- advanced monitoring และ alerting
- automated database migration strategy ระดับองค์กร
- secret manager ของ platform ที่ deploy จริง

## Checkpoint สุดท้าย

เมื่อทำครบเล่ม คุณควรมี ASP.NET Core Web API ที่ประกอบด้วย:

- Controller-based REST API
- EF Core และ SQL Server
- validation และ global error handling
- register/login
- password hashing
- JWT authentication
- role-based authorization
- admin user management
- audit log
- pagination/filtering/sorting
- logging/configuration/OpenAPI
- unit test และ integration test
- Dockerfile และ Docker Compose
- checklist ก่อน deploy

ถ้าข้อเหล่านี้ผ่านและคำสั่ง build/test/publish/container ผ่านจริง โปรเจกต์นี้ถือว่าพร้อมส่งงานหรือใช้เป็นฐานสำหรับ production hardening ภาคถัดไป
