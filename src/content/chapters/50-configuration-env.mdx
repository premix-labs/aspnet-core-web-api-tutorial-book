Configuration คือค่าที่เปลี่ยนได้ตาม environment เช่น connection string, JWT issuer, JWT signing key และ logging level

ASP.NET Core อ่าน configuration ได้จากหลายแหล่ง เช่น `appsettings.json`, environment variables, command-line arguments และ user secrets

## วิธีเรียนบทนี้

บทนี้จะทำให้ project ไม่ผูก secret ไว้กับ source code:

1. แยกว่าค่าไหนเป็น configuration
2. แยกว่าค่าไหนเป็น secret
3. ตั้ง environment variables แบบ Windows และ macOS/Linux
4. ใช้ user secrets สำหรับ local development
5. ตรวจว่า app fail เร็วเมื่อ config สำคัญหาย
6. ปรับ `appsettings.json` ให้ไม่มี secret

## สิ่งที่จะใช้ในบทนี้

| สิ่งที่จะใช้ | ความหมาย |
| --- | --- |
| `appsettings.json` | ค่า config กลางที่ commit ได้ถ้าไม่ใช่ secret |
| environment variable | ค่าที่ host หรือ shell ส่งให้ application ตอน runtime |
| `__` | ตัวแทน `:` ในชื่อ environment variable |
| user secrets | secret local สำหรับ development ที่ไม่ถูก commit |
| fail fast | config สำคัญหายแล้ว app หยุดตั้งแต่ start |

## ค่าใดควรเป็น configuration

- database connection string
- JWT issuer
- JWT audience
- JWT signing key
- token expiration
- logging level
- feature flags
- URL ของ service ภายนอก

หลักคิดง่าย ๆ คือถ้าค่านั้นอาจต่างกันระหว่างเครื่อง dev, staging และ production ให้ใส่ใน configuration

## ค่าใดไม่ควร commit ลง Git

- production database password
- JWT signing key จริง
- API key ของ service ภายนอก
- credential ของ cloud provider
- private certificate หรือ private key

ไฟล์ตัวอย่างใน repository ควรมีเฉพาะค่าที่ไม่ใช่ secret หรือ placeholder ส่วนค่า development ที่เป็น secret ให้เก็บใน user secrets หรือ `.env` ที่ไม่ commit

## ขั้นที่ 1: เข้าใจชื่อ key แบบ nested

ใน `appsettings.json` เราเขียน nested object ได้แบบนี้:

```json
{
  "Jwt": {
    "SigningKey": "value"
  }
}
```

เวลาใช้ environment variable ต้องเขียนเป็น:

```text
Jwt__SigningKey
```

เครื่องหมาย `__` ใช้แทน `:` ใน configuration path ดังนั้น `Jwt__SigningKey` จะ map ไปที่ `Jwt:SigningKey`

## ขั้นที่ 2: ตั้ง environment variables บน Windows

รันใน Windows PowerShell จากโฟลเดอร์ `Backend.Api`

```powershell
$env:LOCAL_SQL_PASSWORD="Replace_With_Strong_Local_Password_123!"
$env:Jwt__SigningKey="replace-with-local-development-signing-key-at-least-32-bytes"
$env:ConnectionStrings__DefaultConnection="Server=localhost,1433;Database=BackendApiDb;User Id=sa;Password=$($env:LOCAL_SQL_PASSWORD);TrustServerCertificate=True;"
```

ค่าเหล่านี้อยู่เฉพาะ session ของ terminal นี้ ถ้าปิด terminal แล้วต้องตั้งใหม่

## ขั้นที่ 3: ตั้ง environment variables บน macOS/Linux

รันใน Bash จากโฟลเดอร์ `Backend.Api`

```bash
export LOCAL_SQL_PASSWORD='Replace_With_Strong_Local_Password_123!'
export Jwt__SigningKey='replace-with-local-development-signing-key-at-least-32-bytes'
export ConnectionStrings__DefaultConnection="Server=localhost,1433;Database=BackendApiDb;User Id=sa;Password=${LOCAL_SQL_PASSWORD};TrustServerCertificate=True;"
```

Bash ใช้ `export` ส่วน PowerShell ใช้ `$env:`

## ขั้นที่ 4: ใช้ user secrets สำหรับ local development

User secrets เหมาะกับค่า local ที่ไม่อยากตั้งใหม่ทุกครั้งที่เปิด terminal

รันจากโฟลเดอร์ `Backend.Api`

```powershell
dotnet user-secrets init
dotnet user-secrets set "Jwt:SigningKey" "replace-with-local-development-signing-key-at-least-32-bytes"
```

ตั้ง connection string:

```powershell
dotnet user-secrets set "ConnectionStrings:DefaultConnection" "Server=localhost,1433;Database=BackendApiDb;User Id=sa;Password=Replace_With_Strong_Local_Password_123!;TrustServerCertificate=True;"
```

User secrets ไม่ได้ encrypt secret แต่ช่วยไม่ให้ secret ถูก commit เข้า repository

## ขั้นที่ 5: Validate configuration ตอน start

ในบท JWT เราเช็ก signing key ด้วย code แล้ว:

```csharp
if (string.IsNullOrWhiteSpace(jwtOptions.SigningKey) ||
    jwtOptions.SigningKey.Length < 32)
{
    throw new InvalidOperationException(
        "Jwt signing key must be at least 32 characters.");
}
```

แนวคิดสำคัญคือถ้า config สำคัญหาย application ควร fail ตอน start ไม่ใช่รอไป error ตอนมี request จริง

## ขั้นที่ 6: ปรับ appsettings.json ให้ปลอดภัยขึ้น

ใน `appsettings.json` ให้เก็บค่า non-secret เช่น:

```json
{
  "Jwt": {
    "Issuer": "Backend.Api",
    "Audience": "Backend.ApiClient",
    "ExpirationMinutes": 60
  }
}
```

สังเกตว่าไม่มี `SigningKey` และไม่มี connection string จริงในไฟล์นี้ ค่าเหล่านี้ให้มาจาก user secrets, environment variable หรือ secret store แทน

## ขั้นที่ 7: ทดสอบว่า config ถูกอ่านจริง

รันจากโฟลเดอร์ `Backend.Api`

```powershell
dotnet run
```

ถ้า app start ได้ แปลว่าค่า config สำคัญถูกส่งเข้ามาแล้ว

ถ้า app error เรื่อง `Jwt signing key` หรือ connection string ให้ย้อนตรวจ environment variable หรือ user secrets ก่อน

## Checkpoint

ก่อนอ่านบทต่อไป ให้ตรวจว่าทำได้ครบตามนี้

- เข้าใจว่า `__` ใน environment variable map เป็น `:`
- ย้าย `Jwt:SigningKey` ออกจาก source code ได้
- ใช้ `dotnet user-secrets` สำหรับ local secret ได้
- ไม่ commit production secret ลง Git
- application fail เร็วถ้า config สำคัญหาย
