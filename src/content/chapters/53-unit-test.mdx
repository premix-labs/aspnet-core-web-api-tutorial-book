Unit test ใช้ทดสอบ logic ขนาดเล็กโดยไม่ต้องรัน application ทั้งตัวหรือเชื่อม database จริง

ในโปรเจกต์นี้สิ่งที่เหมาะกับ unit test ได้แก่ role validation, pagination calculation, mapping และ business rule ที่ไม่ต้องแตะ database

## วิธีเรียนบทนี้

บทนี้จะเริ่มจาก test ที่เล็กและเข้าใจง่าย:

1. สร้าง test project ด้วย xUnit
2. เพิ่ม test project เข้า solution
3. reference project API
4. เขียน test สำหรับ `Roles.IsValid`
5. รัน `dotnet test`
6. แยกว่าอะไรควรเป็น unit test และอะไรควรเป็น integration test

## สิ่งที่จะใช้ในบทนี้

| สิ่งที่จะใช้ | ความหมาย |
| --- | --- |
| xUnit | test framework ที่ใช้ในหนังสือ |
| `[Fact]` | test case ที่ไม่มี parameter |
| `[Theory]` | test case ที่รับหลายชุดข้อมูล |
| `[InlineData]` | ชุดข้อมูลที่ส่งเข้า theory |
| `Assert` | คำสั่งตรวจผลลัพธ์ของ test |
| project reference | ให้ test project เรียก code ใน API project ได้ |

## หลังจบบทนี้ ไฟล์ที่เปลี่ยน

```text
Backend.Api.Tests/Backend.Api.Tests.csproj
Backend.Api.Tests/RolesTests.cs
Backend.Api.sln หรือ Backend.Api.slnx
```

## ขั้นที่ 1: สร้าง test project

รันจาก root ของ solution คือโฟลเดอร์ที่มี `Backend.Api` อยู่

```powershell
dotnet new xunit -n Backend.Api.Tests -f net10.0
```

คำสั่งนี้สร้าง project สำหรับ test ชื่อ `Backend.Api.Tests`

## ขั้นที่ 2: เพิ่ม test project เข้า solution

ถ้ามี solution แล้ว ให้รัน:

Windows PowerShell:

```powershell
dotnet sln add Backend.Api.Tests\Backend.Api.Tests.csproj
```

macOS/Linux Bash:

```bash
dotnet sln add Backend.Api.Tests/Backend.Api.Tests.csproj
```

ใน .NET รุ่นใหม่ไฟล์ solution อาจเป็น `.slnx` ได้ คำสั่ง `dotnet sln add ...` ยังใช้ได้ถ้าอยู่ในโฟลเดอร์เดียวกับไฟล์ solution นั้น

ถ้ายังไม่มี solution ให้สร้างก่อน:

```powershell
dotnet new sln -n Backend.Api
dotnet sln add Backend.Api\Backend.Api.csproj
```

## ขั้นที่ 3: Reference API project

ให้ test project อ้างอิง API project

Windows PowerShell:

```powershell
dotnet add Backend.Api.Tests\Backend.Api.Tests.csproj reference Backend.Api\Backend.Api.csproj
```

macOS/Linux Bash:

```bash
dotnet add Backend.Api.Tests/Backend.Api.Tests.csproj reference Backend.Api/Backend.Api.csproj
```

ถ้าไม่ reference project API ไฟล์ test จะใช้ `Backend.Api.Constants.Roles` ไม่ได้

## ขั้นที่ 4: สร้าง RolesTests.cs

รันจาก root ของ solution

Windows PowerShell:

```powershell
New-Item -ItemType File -Force -Path Backend.Api.Tests/RolesTests.cs
```

macOS/Linux Bash:

```bash
touch Backend.Api.Tests/RolesTests.cs
```

เปิดไฟล์:

```text
Backend.Api.Tests/RolesTests.cs
```

เพิ่ม using และ class:

```csharp
using Backend.Api.Constants;

namespace Backend.Api.Tests;

public class RolesTests
{
}
```

## ขั้นที่ 5: เพิ่ม test role ที่ถูกต้อง

เพิ่ม test นี้ใน `RolesTests`

```csharp
[Theory]
[InlineData(Roles.User)]
[InlineData(Roles.Admin)]
[InlineData("user")]
[InlineData("admin")]
public void IsValid_WhenRoleIsKnown_ReturnsTrue(string role)
{
    var result = Roles.IsValid(role);

    Assert.True(result);
}
```

`[Theory]` ทำให้ method เดียวรันหลายรอบตามค่า `[InlineData]`

## ขั้นที่ 6: เพิ่ม test role ที่ไม่รู้จัก

เพิ่ม test นี้ต่อจาก test แรก:

```csharp
[Theory]
[InlineData("")]
[InlineData("SuperAdmin")]
[InlineData("Manager")]
public void IsValid_WhenRoleIsUnknown_ReturnsFalse(string role)
{
    var result = Roles.IsValid(role);

    Assert.False(result);
}
```

ชื่อ test ควรอ่านแล้วรู้เงื่อนไขและผลที่คาดหวัง เช่น `WhenRoleIsUnknown_ReturnsFalse`

## ขั้นที่ 7: รัน test

รันจาก root ของ solution

```powershell
dotnet test
```

ผลลัพธ์ที่คาดหวังคือ test ผ่านทั้งหมด

ถ้า test compile ไม่ผ่าน ให้ตรวจว่า:

- test project reference API project แล้ว
- namespace ของ `Roles` ถูกต้อง
- `Roles.User` และ `Roles.Admin` มีอยู่จริง

## ควร test อะไรเพิ่ม

- `Roles.IsValid`
- logic ป้องกัน admin demote/deactivate ตัวเอง
- validation helper ที่ไม่ต้องใช้ database
- mapping จาก entity เป็น response DTO
- pagination metadata เช่น `TotalPages`

## ไม่ควร unit test อะไร

ไม่ควรพยายาม unit test EF Core query หนัก ๆ ด้วย mock ที่ซับซ้อนเกินไป เพราะจะกลายเป็น test implementation detail

ถ้าต้องทดสอบ endpoint, middleware, auth หรือ database behavior ให้ใช้ integration test ในบทถัดไป

## Checkpoint

ก่อนอ่านบทต่อไป ให้ตรวจว่าทำได้ครบตามนี้

- มี test project
- project test reference project API
- มี `RolesTests`
- รัน `dotnet test` ผ่าน
- แยกได้ว่าอะไรควรเป็น unit test และอะไรควรเป็น integration test
