# ทำ CRUD กับฐานข้อมูลจริง

ก่อนหน้านี้เราใช้ list ใน memory เพื่อเรียน Controller, Service, Repository และ DTO แต่ข้อมูลแบบนั้นจะหายทุกครั้งที่ restart application

บทนี้จะเปลี่ยน repository จาก `InMemoryUserRepository` ไปเป็น `UserRepository` ที่ใช้ SQL Server ผ่าน EF Core โดยยังรักษา flow เดิมไว้:

```text
UsersController
  -> IUserService
  -> IUserRepository
  -> UserRepository
  -> AppDbContext
  -> SQL Server
```

## วิธีเรียนบทนี้

บทนี้มีหลายไฟล์ ให้ทำเป็นรอบสั้น ๆ และ build หลังแต่ละรอบ:

1. ปรับ `IUserRepository` ให้เป็น async
2. สร้าง `UserRepository` ที่ใช้ EF Core
3. เปลี่ยน DI จาก `InMemoryUserRepository` เป็น `UserRepository`
4. ลบ `InMemoryUserRepository` เดิมออกจาก project
5. ปรับ `UserResponse` ให้ตรงกับ entity ใหม่
6. ปรับ `IUserService` และ `UserService` ให้เป็น async
7. ปรับ `UsersController` ให้ await service
8. ทดสอบว่า restart app แล้วข้อมูลยังอยู่

อย่าคัดลอก code ทั้งบททีเดียว ถ้า build fail จะหาจุดผิดยากมาก

## ก่อนเริ่มบทนี้

ให้ตรวจว่าคุณทำบทก่อนหน้าครบแล้ว:

- มี `Models/User.cs`
- มี `Data/AppDbContext.cs`
- `Program.cs` ลงทะเบียน `AddDbContext`
- สร้าง migration และรัน `dotnet tool run dotnet-ef database update` แล้ว
- โครงสร้างจากภาค Architecture มี `Controllers`, `Services`, `Repositories` และ DTO พื้นฐานแล้ว

ถ้าจะรันคำสั่ง ให้เปิด terminal ที่ root ของโปรเจกต์ `Backend.Api`

## คำศัพท์ในบทนี้

| สิ่งที่จะใช้ | ความหมาย |
| --- | --- |
| `Task<T>` | ผลลัพธ์ของงาน async ที่จะได้ค่า `T` ในอนาคต |
| `async` | บอกว่า method นี้มีงาน asynchronous ภายใน |
| `await` | รอผลลัพธ์ของงาน async โดยไม่ block thread |
| `ToListAsync()` | อ่านข้อมูลจาก database เป็น list แบบ async |
| `FirstOrDefaultAsync(...)` | หา record แรกที่ตรงเงื่อนไข ถ้าไม่พบคืน `null` |
| `SaveChangesAsync()` | บันทึกการเปลี่ยนแปลงลง database |
| `AsNoTracking()` | อ่านข้อมูลแบบไม่ให้ EF Core track entity |

## ขั้นที่ 1: ปรับ Repository Interface

เปิดไฟล์:

```text

```

เปลี่ยน method จาก synchronous เป็น asynchronous:

```csharp
using Backend.Api.Models;

namespace Backend.Api.Repositories;

public interface IUserRepository
{
    Task<List<User>> GetAllAsync();
    Task<User?> GetByIdAsync(int id);
    Task<User?> GetByEmailAsync(string email);
```

เหตุผลที่ต้องเปลี่ยนเป็น async เพราะ database query เป็นงาน I/O การใช้ `Task` และ `await` ช่วยให้ server ไม่ต้อง block thread ระหว่างรอ SQL Server ตอบกลับ

## ขั้นที่ 3: เพิ่ม method อ่านข้อมูล

เพิ่ม method เหล่านี้ใน class `UserRepository`

```csharp
public Task<List<User>> GetAllAsync()
{
    return db.Users
        .AsNoTracking()
        .OrderBy(user => user.Id)
        .ToListAsync();
}

public Task<User?> GetByIdAsync(int id)
{
    return db.Users.FirstOrDefaultAsync(user => user.Id == id);
}
```

อ่าน code นี้ทีละส่วน:

- `db.Users` คือ table `Users`
- `AsNoTracking()` ใช้กับ query ที่อ่านอย่างเดียว
- `OrderBy(user => user.Id)` ทำให้ผลลัพธ์เรียงลำดับคาดเดาได้
- `ToListAsync()` ส่ง query ไป database แล้วคืน list
- `FirstOrDefaultAsync(...)` คืน user ตัวแรกที่ตรงเงื่อนไข หรือ `null`

## ขั้นที่ 5: เพิ่ม method เขียนข้อมูล

เพิ่ม method เหล่านี้ต่อจาก `GetByEmailAsync`

```csharp
public async Task<User> CreateAsync(User user)
{
    db.Users.Add(user);
    await db.SaveChangesAsync();

    return user;
}
```

`db.Users.Add(user)` ทำให้ EF Core รู้ว่ามี entity ใหม่ต้อง insert ส่วน `SaveChangesAsync()` คือจังหวะที่ส่งคำสั่งไป database จริง

เพิ่ม update:

```csharp
public async Task<bool> UpdateAsync(User user)
{
    user.UpdatedAtUtc = DateTime.UtcNow;
    await db.SaveChangesAsync();

    return true;
}
```

เราไม่ต้องเรียก `db.Users.Update(user)` ในกรณีนี้ เพราะ user ถูกอ่านมาจาก `GetByIdAsync` และ EF Core track entity นั้นอยู่แล้ว

เพิ่ม delete:

```csharp
public async Task<bool> DeleteAsync(User user)
{
    db.Users.Remove(user);
    await db.SaveChangesAsync();

    return true;
}
```

ถ้า build error ว่า `UserRepository` ยัง implement interface ไม่ครบ ให้ตรวจว่ามีครบทั้ง 6 method ที่ลงท้ายด้วย `Async`

## ขั้นที่ 6: เปลี่ยน DI Registration

## ขั้นที่ 6.1: ลบ InMemoryUserRepository เดิมออก

หลังจากเปลี่ยน `IUserRepository` เป็น async แล้ว ไฟล์ `InMemoryUserRepository.cs` เดิมจากภาค Architecture จะ compile ไม่ผ่าน เพราะ class นั้นยัง implement method ชุดเก่า เช่น `GetAll()` และ `GetById()` แต่ interface ใหม่ต้องการ `GetAllAsync()` และ `GetByIdAsync(...)`

ในบทนี้เราจะไม่ใช้ in-memory repository แล้ว ให้ลบไฟล์นี้ออกจาก project:

```text
Repositories/InMemoryUserRepository.cs
```

Windows PowerShell:

```powershell
Remove-Item -LiteralPath Repositories/InMemoryUserRepository.cs -Force
```

macOS/Linux Bash:

```bash
rm -f Repositories/InMemoryUserRepository.cs
```

ถ้าไม่ลบไฟล์นี้ คุณอาจเจอ error ประมาณนี้ตอน `dotnet build`:


error นี้ไม่ได้แปลว่า `UserRepository` ใหม่ผิด แต่แปลว่า project ยัง compile class เก่าที่ไม่ตรงกับ interface ใหม่อยู่

## ขั้นที่ 10: เพิ่ม method อ่านข้อมูลใน UserService

เพิ่ม method เหล่านี้ใน class `UserService`

```csharp
public async Task<IReadOnlyList<UserResponse>> GetUsersAsync()
{
    var users = await userRepository.GetAllAsync();

    return users
        .Select(ToResponse)
        .ToList();
}

public async Task<UserResponse?> GetUserByIdAsync(int id)
{
    var user = await userRepository.GetByIdAsync(id);

    return user is null ? null : ToResponse(user);
}
```

Controller จะยังใช้ `null` จาก service เพื่อตัดสินใจตอบ `404 Not Found`

## ขั้นที่ 11: เพิ่ม method สร้างและแก้ไข user

เพิ่ม method สร้าง user:

```csharp
public async Task<UserResponse> CreateUserAsync(CreateUserRequest request)
{
    var existingUser = await userRepository.GetByEmailAsync(request.Email);

    if (existingUser is not null)
    {
        throw new InvalidOperationException("Email already exists.");
    }

    var user = new User
    {
        Email = request.Email,
        PasswordHash = "pending-auth",
        Role = "User",
        IsActive = true
    };

    var createdUser = await userRepository.CreateAsync(user);

    return ToResponse(createdUser);
}
```

`pending-auth` เป็น placeholder ชั่วคราวเพื่อให้ column `PasswordHash` ที่เป็น required field มีค่าในช่วงที่ยังไม่ได้เรียน password hashing

ค่านี้ใช้ login ไม่ได้ และไม่ใช่ pattern สำหรับ production เมื่อถึงภาค Authentication เราจะเปลี่ยนเป็น hash ที่สร้างจาก password จริงด้วย password hasher

เพิ่ม method แก้ไข:

```csharp
public async Task<UserResponse?> UpdateUserAsync(int id, UpdateUserRequest request)
{
    var user = await userRepository.GetByIdAsync(id);

    if (user is null)
    {
        return null;
    }

    user.Email = request.Email;

    await userRepository.UpdateAsync(user);

    return ToResponse(user);
}
```

ถ้าไม่พบ user ให้คืน `null` เพื่อให้ Controller ตอบ `404 Not Found`

## ขั้นที่ 15: แก้ POST และ PUT actions

แก้ `POST` ให้เรียก service แบบ async:

```csharp
[HttpPost]
public async Task<IActionResult> CreateUser(CreateUserRequest request)
{
    try
    {
        var user = await userService.CreateUserAsync(request);

        return CreatedAtAction(nameof(GetUserById), new { id = user.Id }, user);

```

`Conflict(...)` ใช้กับกรณี email ซ้ำ เพราะ request นี้ชนกับข้อมูลที่มีอยู่แล้ว

```csharp
}
```

## ทดสอบ API

รัน API:

```powershell
dotnet run
```

ให้ใช้ `baseUrl` ตาม port จริงที่เครื่องคุณแสดง ตัวอย่าง:

```http
@baseUrl = http://localhost:<http-port>

### Create user in database
POST {{baseUrl}}/api/v1/users
Content-Type: application/json

{
  "email": "database-user@example.com"
}

### Get all users
GET {{baseUrl}}/api/v1/users
Accept: application/json
```

ถ้าเครื่องของคุณใช้ HTTPS และ certificate พร้อมแล้ว จะเปลี่ยนเป็น `https://localhost:<https-port>` ก็ได้

ให้ตรวจผลลัพธ์ตามตารางนี้:

| Request | ผลลัพธ์ที่ควรได้ |
| --- | --- |
| `POST /api/v1/users` ด้วย email ใหม่ | `201 Created` และ body เป็น `UserResponse` |
| `POST /api/v1/users` ด้วย email เดิม | `409 Conflict` |
| `GET /api/v1/users` | `200 OK` และมีข้อมูลจาก database |
| `GET /api/v1/users/999` | `404 Not Found` |
| `PUT /api/v1/users/1` | `200 OK` ถ้า id `1` มีอยู่ |
| `PUT /api/v1/users/999` | `404 Not Found` |
| `DELETE /api/v1/users/999` | `404 Not Found` |
| `DELETE /api/v1/users/1` | `204 No Content` ถ้า id `1` มีอยู่ |

ถ้า database ของคุณไม่ได้มี user id `1` ให้ใช้ id ที่ได้จาก response ของ `POST /api/v1/users` แทน

จากนั้น restart application แล้วเรียก `GET /api/v1/users` อีกครั้ง ข้อมูลที่สร้างไว้ควรยังอยู่ เพราะตอนนี้ถูกเก็บใน database แล้ว ไม่ได้อยู่ใน memory list แล้ว

## ข้อผิดพลาดที่เจอบ่อย

ถ้า error ว่า `Cannot insert the value NULL into column PasswordHash` แปลว่าสร้าง `User` โดยไม่ได้ใส่ `PasswordHash`

ถ้า email ซ้ำแล้วได้ error จาก database แปลว่ายังไม่ได้ตรวจ `GetByEmailAsync` ก่อน insert หรือไม่ได้ catch error เป็น `409 Conflict`

ถ้า endpoint ยังได้ข้อมูลเดิมแบบ reset หลัง restart ให้ตรวจว่า `Program.cs` ยังลงทะเบียน `InMemoryUserRepository` อยู่หรือไม่

ถ้า route ใน `.http` ไม่ตรงกับ `[Route("api/v1/[controller]")]` จะได้ `404 Not Found` ให้ตรวจว่า request ใช้ `/api/v1/users`

และไม่ควรมี registration นี้เหลืออยู่:

```csharp
builder.Services.AddScoped<IUserRepository, InMemoryUserRepository>();
```

## Checkpoint

ก่อนอ่านบทต่อไป ให้ตรวจว่าทำได้ครบตามนี้

- `IUserRepository` เป็น async แล้ว
- มี `UserRepository` ที่ใช้ `AppDbContext`
- ลบ `Repositories/InMemoryUserRepository.cs` แล้ว หรืออย่างน้อยไม่มี class นี้ถูก compile อยู่ใน project
- `Program.cs` ลงทะเบียน `IUserRepository` เป็น `UserRepository`
- `UserResponse` ไม่มี `PasswordHash`
- `UserService` ใช้ async/await กับ repository
- `UsersController` ยังเรียกผ่าน `IUserService`
- `POST` email ซ้ำตอบ `409 Conflict`
- สร้าง user แล้ว restart application ข้อมูลไม่หาย
